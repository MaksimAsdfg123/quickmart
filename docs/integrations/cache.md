# Стратегия кэширования

## 1. Назначение

Кэширование добавлено для снижения повторной нагрузки на read-heavy публичные сценарии каталога без изменения бизнес-поведения checkout, корзины, заказов и складского контура.

Решение ускоряет:

- повторное чтение страниц публичного каталога;
- повторное чтение карточки товара;
- повторное чтение публичного списка категорий.

Основная цель внедрения состоит в сокращении числа обращений к базе данных и репозиториям на наиболее частых сценариях чтения при сохранении предсказуемой инвалидации после изменений товара, категории и складского остатка.

## 2. Анализ кандидатов на кэширование

### 2.1 Рассмотренные кандидаты

| Кандидат | Характер нагрузки | Выбор | Обоснование |
| --- | --- | --- | --- |
| `ProductService.getCatalog(...)` | Частые публичные чтения, пагинация, маппинг карточек товара | Выбран | Наиболее очевидный read-heavy сценарий. На cache miss выполняется запрос каталога и дополнительное чтение остатка для каждой карточки товара. |
| `ProductService.getProduct(id)` | Частые публичные чтения карточки товара | Выбран | Данные товара читаются существенно чаще, чем изменяются. Ответ содержит `availableQuantity`, поэтому необходима инвалидация при изменении stock. |
| `CategoryService.getPublicCategories()` | Справочные данные, редко меняются | Выбран | Небольшой, часто переиспользуемый reference dataset с простой и надежной инвалидацией. |
| `DeliverySlotService.getUpcomingSlots()` | Read-heavy, но ниже частота и выше чувствительность к операционным изменениям | Не выбран | Сценарий подходит для следующего этапа, но его производительный эффект ниже, чем у публичного каталога. |
| Корзина, checkout, order history, inventory admin APIs | Высокая чувствительность к актуальности | Не выбран | Для этих сценариев риск stale data выше, чем потенциальная выгода от кэша. |
| Промокоды и расчеты checkout | Критична консистентность и корректность бизнес-правил | Не выбран | Кэширование здесь способно исказить поведение checkout и усложнить валидацию. |

### 2.2 Причины окончательного выбора

Выбран небольшой, но архитектурно оправданный набор read-моделей:

- публичные страницы каталога;
- карточки товаров;
- публичный список категорий.

Именно эти сценарии одновременно удовлетворяют инженерным критериям:

- имеют высокую долю чтений;
- редко изменяются относительно частоты чтения;
- допускают явную и детерминированную инвалидацию;
- не требуют персонализированных ключей;
- не влияют на критичную транзакционную логику checkout.

## 3. Принятый подход

### 3.1 Cache strategy

Используется подход `cache-aside` на сервисном слое.

Схема работы:

1. Публичный read-метод вызывается через `Spring Cache`.
2. При cache hit ответ возвращается из in-memory кэша.
3. При cache miss выполняется обычная бизнес-логика и результат сохраняется в кэш.
4. После изменения связанных данных кэш инвалидируется через доменные invalidation events.

Такой подход выбран как наиболее прагматичный для текущего монолитного backend:

- не требует перестройки текущих сервисов;
- не добавляет внешнюю инфраструктуру;
- сохраняет прозрачную точку контроля инвалидации;
- легко отключается конфигурацией.

### 3.2 Cache provider

Реализация построена на `Spring Cache` и `Caffeine`.

Причины выбора:

- стек нативно поддерживается Spring Boot;
- нет внешней сетевой зависимости;
- можно задать разные TTL и `maximumSize` по cache region;
- решение уместно для текущего single-instance/local-first масштаба проекта.

### 3.3 Scope

Кэш применяется только к публичным read-операциям:

- `ProductService.getCatalog(...)`;
- `ProductService.getProduct(id)`;
- `CategoryService.getPublicCategories()`.

Административные списки, checkout, корзина, складские операции и ордерный контур намеренно не кэшируются.

## 4. Что кэшируется

| Cache region | Метод | Данные | TTL | Maximum size |
| --- | --- | --- | --- | --- |
| `publicCatalogPages` | `ProductService.getCatalog(...)` | Пагинированный публичный каталог | `30s` | `256` |
| `publicProductCards` | `ProductService.getProduct(id)` | Карточка товара | `30s` | `1024` |
| `publicCategories` | `CategoryService.getPublicCategories()` | Публичный список категорий | `30m` | `64` |

### 4.1 Почему TTL различается

- Каталог и карточки товаров зависят от складского остатка и названия категории, поэтому TTL выбран коротким: `30s`.
- Публичные категории меняются заметно реже, поэтому TTL выбран длиннее: `30m`.

TTL не является основной защитой от устаревания данных. Основной механизм актуализации — явная инвалидация после изменения данных. TTL используется как дополнительный safeguard и как ограничение жизненного цикла записей.

## 5. Схема ключей

Формирование ключей централизовано в `CacheKeyFactory`.

### 5.1 Правила

- ключ всегда содержит scope региона;
- для каталога учитываются все параметры, влияющие на ответ;
- строковые параметры нормализуются;
- для singleton cache используется отдельный предсказуемый префикс.

### 5.2 Форматы ключей

- каталог:
  `catalog:category=<uuid|all>:query=<normalized|_>:page=<n>:size=<n>`
- карточка товара:
  `product:<productId>`
- список категорий:
  `scope:public-categories`

### 5.3 Примеры

- `catalog:category=all:query=_:page=0:size=12`
- `catalog:category=30000000-0000-0000-0000-000000000001:query=milk:page=0:size=12`
- `product:40000000-0000-0000-0000-000000000001`
- `scope:public-categories`

## 6. Инвалидация

### 6.1 Общий принцип

Инвалидация выполнена не через точечные аннотации по всему коду, а через отдельный infrastructure layer:

- `CatalogReadCacheInvalidationPublisher`
- `CatalogReadCacheInvalidationListener`
- `ProductReadCacheInvalidationEvent`
- `CategoryReadCacheInvalidationEvent`

Сервисный слой публикует invalidation event внутри транзакции. Фактический сброс кэша происходит в `@TransactionalEventListener(phase = AFTER_COMMIT)`.

Это означает, что кэш не очищается до тех пор, пока транзакция изменения данных не будет успешно зафиксирована.

### 6.2 Правила инвалидации

#### Изменение товара

Источники:

- `ProductService.create(...)`
- `ProductService.update(...)`
- `ProductService.setActive(...)`
- `ProductService.deactivate(...)`

Действия:

- точечно удаляется карточка измененного товара из `publicProductCards`;
- полностью очищается `publicCatalogPages`.

Причина: товар может входить в несколько страниц каталога, а точечная инвалидация всех затронутых страниц без отдельного secondary index в текущем масштабе неоправданно усложнит решение.

#### Изменение остатка

Источники:

- `InventoryService.decreaseStockWithLock(...)`
- `InventoryService.increaseStockWithLock(...)`
- `InventoryService.updateStock(...)`

Действия:

- точечно удаляется карточка соответствующего товара;
- полностью очищается `publicCatalogPages`.

Причина: `availableQuantity` присутствует и в карточке товара, и в каталоге.

#### Изменение категории

Источники:

- `CategoryService.create(...)`
- `CategoryService.update(...)`
- `CategoryService.setActive(...)`
- `CategoryService.delete(...)`

Действия:

- полностью очищается `publicCategories`;
- полностью очищается `publicProductCards`;
- полностью очищается `publicCatalogPages`.

Причина: имя и доступность категории напрямую влияют на публичный список категорий и на поле `categoryName` в ответах каталога и карточек товара.

## 7. Консистентность данных

### 7.1 Модель консистентности

В пределах одного экземпляра приложения используется модель:

- `strong-enough consistency after commit` для поддерживаемых mutation scenarios;
- кратковременная допустимая stale window, ограниченная TTL, если инвалидация не требуется или если запись доживает до естественного истечения.

### 7.2 Как снижен риск stale data

- инвалидация выполняется только после успешного commit;
- ключи нормализованы и детерминированы;
- для данных, зависящих от stock, используется короткий TTL;
- для категорий, влияющих на несколько read-моделей, применяется широкий сброс связанных cache region.

### 7.3 Ограничения текущей реализации

- решение локально для одного инстанса приложения;
- межузловая инвалидация не реализована;
- при горизонтальном масштабировании потребуется либо распределенный cache, либо внешний механизм invalidation events;
- `publicCatalogPages` очищается целиком при любом изменении товара или stock, что сознательно повышает предсказуемость ценой менее точной eviction granularity.

## 8. Поведение при отключении и ошибках

### 8.1 Отключение

При `APP_CACHE_ENABLED=false` используется `NoOpCacheManager`.

Следствия:

- бизнес-логика продолжает работать без изменений;
- все `@Cacheable` методы выполняются как обычные read-запросы;
- кэш не становится обязательной частью runtime.

### 8.2 Недоступность кэша

Текущая реализация использует локальный in-memory cache и не зависит от отдельного сетевого сервиса. Соответственно, отдельный сценарий деградации по типу Redis outage отсутствует.

Если в дальнейшем будет выбран внешний cache backend, потребуется отдельная стратегия graceful degradation и observability для отказов кэша.

## 9. Архитектурные trade-offs

### 9.1 Производительность vs актуальность

- короткий TTL и after-commit invalidation минимизируют stale data;
- более агрессивный TTL увеличил бы cache hit ratio, но повысил бы риск устаревших ответов;
- более короткий TTL дал бы меньше stale window, но и меньший выигрыш по производительности.

### 9.2 Простота vs точность eviction

- для каталога выбрана coarse-grained invalidation: очищается весь region `publicCatalogPages`;
- альтернативой мог бы быть secondary index страниц по товарам и категориям, но для текущего проекта это избыточно.

### 9.3 In-memory cache vs distributed cache

- текущий вариант проще в эксплуатации и быстрее локально;
- распределенный cache оправдан только при multi-instance deployment и подтвержденной необходимости shared cache coherence.

## 11. Тестирование

Автотесты реализованы в `tests/backend` и следуют общему стилю `suites + shared`.

Покрытие включает:

- повторное чтение каталога использует кэш;
- повторное чтение карточки товара использует кэш;
- повторное чтение списка категорий использует кэш;
- изменение товара инвалидирует кэш страниц каталога;
- изменение stock инвалидирует кэш карточки товара и страниц каталога;
- изменение категории инвалидирует связанный product cache;
- деактивация категории инвалидирует список публичных категорий;
- режим `APP_CACHE_ENABLED=false` не ломает поведение и отключает cache hits.

Ключевые suite-файлы:

- `tests/backend/suites/kotlin/com/quickmart/test/suites/cache/catalog/PublicCatalogCacheTest.kt`
- `tests/backend/suites/kotlin/com/quickmart/test/suites/cache/category/PublicCategoryCacheTest.kt`
- `tests/backend/suites/kotlin/com/quickmart/test/suites/cache/config/CacheDisabledTest.kt`

Отдельная задача запуска:

- `.\gradlew.bat :api-tests:cacheTest`

Важно: `cacheTest` пока не запускается отдельным job'ом в стандартном GitHub Actions PR/main pipeline, поэтому при изменениях cache-слоя его нужно прогонять явно.

## 12. Локальный запуск и проверка

Дополнительная инфраструктура не требуется. Кэш локальный и поднимается вместе с backend.

### 12.1 Основные команды

- `.\gradlew.bat :backend:bootRun`
- `.\gradlew.bat :backend:bootJar`
- `.\gradlew.bat :api-tests:cacheTest`

### 12.2 Ручная проверка

1. Запустить backend.
2. Вызвать публичный каталог или карточку товара несколько раз.
3. Изменить товар, категорию или остаток через backend.
4. Повторно запросить те же данные и убедиться, что ответ отражает актуальное состояние.
5. При необходимости повторить сценарий с `APP_CACHE_ENABLED=false`.

## 13. Возможности дальнейшего развития

Следующие кандидаты для следующего этапа:

- `DeliverySlotService.getUpcomingSlots()` при подтвержденной read-нагрузке;
- распределенный cache backend при multi-instance deployment;
- более точная page-level invalidation каталога при появлении требований к высокой cache locality;
- batch/read-model оптимизация для получения `availableQuantity` на cache miss.

Расширять кэширование рекомендуется только после измерений и только для сценариев с предсказуемой инвалидацией.
