# Функциональный обзор

## Назначение

Документ фиксирует фактически реализованный функциональный scope Quickmart и разводит:

- возможности, доступные пользователю через web UI;
- возможности, доступные администратору через web UI;
- возможности, доступные только через backend API.

Это важно, потому что backend и frontend развиваются не полностью синхронно: часть административных операций уже реализована в API, но еще не вынесена в SPA.

## Capability map

| Область | Customer UI | Admin UI | Backend API | Примечание |
| --- | --- | --- | --- | --- |
| Регистрация и логин | Да | Да | Да | JWT выдается через `/api/auth/register` и `/api/auth/login` |
| Публичный каталог | Да | Нет | Да | категории, поиск по имени, пагинация, карточка товара |
| Корзина | Да | Нет | Да | защищенный пользовательский flow |
| Адресная книга | Да | Нет | Да | CRUD для адресов текущего пользователя |
| Checkout | Да | Нет | Да | адрес + delivery slot + promo code + payment method |
| История заказов пользователя | Да | Нет | Да | просмотр списка, деталей и отмена заказа |
| Управление товарами | Нет | Да | Да | frontend admin UI и `/api/admin/products` |
| Управление категориями | Нет | Да | Да | frontend admin UI и `/api/admin/categories` |
| Управление остатками | Нет | Да | Да | frontend admin UI и `/api/admin/inventory` |
| Управление заказами | Нет | Да | Да | frontend admin UI и `/api/admin/orders` |
| Управление промокодами | Нет | Да | Да | frontend admin UI и `/api/admin/promocodes` |
| Управление delivery slots | Нет | Нет | Да | реализовано только через `/api/admin/delivery-slots` |
| Kafka audit trail | Нет | Косвенно | Да | данные читаются через `/api/admin/orders/{id}/events`, если Kafka включена |

## Пользовательский поток

### Публичная часть

- просмотр каталога товаров;
- фильтрация по категориям;
- поиск по имени товара;
- просмотр карточки товара.

Публичные catalog endpoints доступны без JWT и дополнительно защищены read-cache в backend.

### Аутентифицированный customer flow

После логина пользователь получает доступ к:

- корзине;
- адресу доставки;
- checkout;
- истории заказов;
- деталям заказа;
- отмене собственного заказа, если transition разрешен backend.

### Checkout rules

Backend сейчас применяет следующие правила оформления заказа:

- пустая корзина не может быть оформлена;
- для checkout обязателен адрес, принадлежащий текущему пользователю;
- delivery slot должен быть активным, не в прошлом и не переполненным;
- заказ списывает остатки по товарам в момент checkout;
- бесплатная доставка включается при subtotal от `1500.00`, иначе применяется фиксированная доставка `149.00`;
- промокод валидируется по активности, сроку действия, лимиту использования и минимальной сумме заказа.

### Payment methods

Поддерживаемые способы оплаты:

- `CARD` - заказ сразу фиксируется как оплаченный;
- `CASH` - до доставки оплата остается `PENDING`;
- `MOCK_ONLINE` - используется как тестовый онлайн-метод; при сумме заказа выше `50000` backend возвращает ошибку оплаты.

## Административный поток

Текущая frontend admin-панель покрывает:

- товары;
- категории;
- остатки;
- заказы;
- промокоды.

Административные endpoints защищены ролью `ADMIN`, а frontend использует отдельные `AdminRoute` и набор страниц в `app/frontend/src/pages/admin`.

### Что доступно только в backend API

Сейчас backend поддерживает административное управление delivery slots, но в SPA нет отдельного экрана для этой функции. Это не ошибка документации, а реальный сдвиг в реализации между API и UI.

## Состояние заказа

Поддерживаемые статусы:

- `CREATED`
- `CONFIRMED`
- `ASSEMBLING`
- `OUT_FOR_DELIVERY`
- `DELIVERED`
- `CANCELLED`

Backend контролирует допустимые переходы между статусами и использует эти переходы как источник Kafka-событий, если интеграция включена.

## Текущие ограничения

- admin delivery slots доступны только в backend API;
- UI automation пока не покрывает catalog/cart/checkout/admin flows, кроме auth/access;
- black-box API automation пока сосредоточена на auth/access contract, а не на полном REST-регрессе всех бизнес-сценариев;
- Kafka audit trail существует только при включенном `APP_KAFKA_ENABLED=true`.
